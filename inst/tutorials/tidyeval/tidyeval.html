Tidyeval is the framework that dplyr uses to adapt the expression to the situation, so that “it just works”.

Things to keep in mind

However, as Jar Jar discovered, there is no such thing as a free lunch. There are a couple things to be mindful of.

Referential transparency

Most arguments to dplyr functions are not referentially transparent. You can’t replace an argument with a seemingly-equivalent argument that you have defined elsewhere.

As you see, neither of these substitutions works how we expect - so it is not obvious how to call dplyr functions without hard-coding everything.

Ambiguity

As well, dplyr evaluates expressions ambiguously. As we will see, the code filter(df, x == y) can be interpreted differently depending on the data and the context:

df[df$x == df$y, ]
df[df$x == y, ]
df[x == df$y, ]
df[x == y, ]

If the “wrong” thing happens when you are working interactively, you can see that there is a problem and correct it. If the “wrong” thing happens when you are calling a function (from a function, from a packge, …), it can be nearly impossible to figure out what went wrong, let alone fix it.

Summary

Tidyeval provides a set of rules for the default evaluation of expressions, and a set of tools that let you change that default whenever you need to. For people who write functions and people who use those functions, tidyeval provides a framework that allows both sets of people be confident that a function will do what they expect it to do.

In the next few sections, we will discuss the principles and concepts of tidyeval, then show how you can use these to call functions that use tidyeval (like those in dplyr). Finally, we will discuss how you can write functions that use the tidyeval framework.

Here, we used an expression, species == "Droid"; it is then filter()’s job to evaluate this expression into values - then use those values to return the corresponding rows of the data frame. It may be useful to think of an expression as a set of instructions.

As we have found out, when dplyr evaluates an expression, its default is to look for variable-names first in the data frame, then in the context from which you called the function. Going forward, we will go through some examples of how you can use tidyeval to build expressions that are evaluated according to your specifications.

You can compose expressions using symbols; in this case, species is a symbol. Let’s say you are building a shiny app where you need to call a dplyr function on a data frame that you upload to the app. When you are writing the app, you cannot possibly know the variable names that will be used at runtime.

It may not be evident how to build expression for dplyr to evaluate. We will see how the tidyeval framework lets you build expressions using symbols that you can create at runtime.

Using functions that use tidyeval

One use for tidyeval is to help you write package functions. Another application is writing shiny apps.

In a shiny app, let’s say you want your user to be able to upload a data-frame, filter on some condition, then make a scatterplot, letting your user specify your axes. Let’s also say that you want to use the tidyverse.

For the filtering step, one challenge may be to compose a filtering condition without knowing what the data-frames’s variables are.

For the visualisation step, ggplot2 does not currently use tidyeval; instead you use the aes_string() function. But, by becoming familiar with tidyeval now, it will be that much easier to use the tidyeval framework in ggplot2 when it arrives.

Let’s pick up the filtering example. As you know, the dplyr::filter() function takes a series of expressions that it evaluates to TRUE or FALSE to determine which rows to return. Let’s say that you have a shiny app where you have:

  • an input that specifies the variable for the filter - this will be a character string
  • an input that specifies the value - this will be a numeric

For the sake of simplicity, let’s say that it is single condition and that we are testing equality. For the sake of even more simplicity, let’s start out by specifying we will use the starwars data; we will filter on birth_year, we will vary only the value.

value <- 19

filter(starwars, birth_year == value)
#> # A tibble: 2 x 13
#>             name height  mass hair_color skin_color eye_color birth_year gender homeworld species
#>            <chr>  <int> <dbl>      <chr>      <chr>     <chr>      <dbl>  <chr>     <chr>   <chr>
#> 1 Luke Skywalker    172    77      blond       fair      blue         19   male  Tatooine   Human
#> 2    Leia Organa    150    49      brown      light     brown         19 female  Alderaan   Human
#> # ... with 3 more variables: films <list>, vehicles <list>, starships <list>

So far so good - filter() looks for birth_year in the data and finds it, then looks for value in the data, does not find it, then finds value in the context.

You will need to use a little imagination because this is a contrived example. Let’s imagine that we don’t know that starwars does not have a column named value. We want to make this evaluation “airtight”, we want to make sure that we use value from our context.

Let’s look back at our string example:

library("glue")

value <- 19

glue("We want rows where the birth_year is {value}.")
#> We want rows where the birth_year is 19.

Recall that we are composing a string by unquoting: {name} is replaced with value of name, using the context from which glue() is called.

Instead of composing strings we need to compose expressions, so Tidyeval provides an unquoting function. The UQ() function acts on expressions in tidyeval like the {} acts on strings in glue(): it evaluates its contents and incorporates the result.

Your turn

Make sure this works:

value <- 19

filter(starwars, birth_year == UQ(value))

To convince yourself that the code above is resistant to mischeif, change both instances of value to birth_date and rerun.

There is another thing we can do here, instead of evaluating the filter statement then printing out the data, we can capture the set of instructions, the expression, and print that instead. To do that, we use the quo() function.

value <- 19

quo(
  filter(starwars, birth_year == UQ(value))
)

Run this code to see what happens. Switch value with birth_year again, if you like.

We see again, this word “quosure”. Again, let’s think about a quosure as an expression (set of instructions) with an attached context (set of materials). We see the instructions (~filter(starwars, birth_year == 19)), and we are given an indication of the materials (local). For now, we can use the quo() function to verify that our instructions are what we expect.

Our next step is to specify the dataframe and the column to filter. Recall that symbols can be used to compose expressions. Tidyeval provides a function to convert strings into symbols, sym().

sym("foo")
#> foo

We can use the sym() function to help build the left-hand-side of our expression.

When we use tidyeval with dplyr functions, dplyr evaluates the expressions that you provide. It can be useful to keep in mind what the dplyr function is evaluating to. filter() evaluates its expressions to logical values - it wants TRUE or FALSE. Some dplyr functions do things a little differently, let’s look at some of those differences.

“selecty” functions

There is a family of functions that use the tidyselect package as their back-end; dplyr::select() is the most prominent. Other “selecty” functions include tidyr::gather() and dplyr::pull(). They all work the same way because they all need to identify a set of columns within an existing data source.

This excellent tidyr article dives into the theory and design choices. There are two big points here.

  1. Selecty functions evaluate its expressions to integers that indicate column-positions in the data source.

  2. It is best-practice to provide expressions that make sense only in the data, not in the surrounding context.

Let’s look at some examples of these principles in action. The usual way to call a select function is just to use the bare column-names, which we now know as symbols.

If we want to insist that we look only in the data for the name (not a bad idea), we can use the .data pronoun:

select(starwars, .data$name, .data$species : .data$starships)

Here’s where things can get potentially difficult. Let’s say that you want to select from a column-position to until the last column. This works:

select(starwars, .data$name, .data$species : ncol(starwars))

But the expression ncol(starwars) refers to the context where starwars is found. Our best-practice is to refer only to the data. We now have two references to starwars in the function call; we would prefer to have only one.

“namey” functions

Some dplyr functions, in addition to evaluating expressions, assign column names. These include mutate(), summarise(), and transmute().

For an example, let’s find the minimum birth_year for each species.

starwars %>%
  group_by(species) %>%
  summarise(birth_year_min = min(birth_year, na.rm = TRUE)) 
#> # A tibble: 38 x 2
#>      species birth_year_min
#>        <chr>          <dbl>
#>  1    Aleena            Inf
#>  2  Besalisk            Inf
#>  3    Cerean             92
#>  4  Chagrian            Inf
#>  5  Clawdite            Inf
#>  6     Droid             15
#>  7       Dug            Inf
#>  8      Ewok              8
#>  9 Geonosian            Inf
#> 10    Gungan             52
#> # ... with 28 more rows

Now I want to parameterise this, in case I want to do the same thing with height. (You are likely suspecting that we should really be making a function to do this - which we will in the next section of this tutorial.)

Here’s one way to do it:

var_to_min <- "birth_year"

var_min <- paste0(var_to_min, "_min")

starwars %>%
  group_by(species) %>%
  summarise(UQ(sym(var_min)) := min(UQ(sym(var_to_min)), na.rm = TRUE)) 
#> # A tibble: 38 x 2
#>      species birth_year_min
#>        <chr>          <dbl>
#>  1    Aleena            Inf
#>  2  Besalisk            Inf
#>  3    Cerean             92
#>  4  Chagrian            Inf
#>  5  Clawdite            Inf
#>  6     Droid             15
#>  7       Dug            Inf
#>  8      Ewok              8
#>  9 Geonosian            Inf
#> 10    Gungan             52
#> # ... with 28 more rows

The first step here is to create a new variable-name by appending "_min" to the variable-name of interest. The second step is to adapt the summarise() expression. Note that it has both a left side (new name) and a right side (value). The good news is that we can use our unquoting tools on both sides. However, the = operator does not support unquoting on the left-side, so we use the := operator from the rlang package - designed just for this purpose.

The use of the := operator is the thing that we need to keep in mind when using “namey” functions.

One other point, because the left-side is going to end up as a name, you can unquote a symbol, UQ(sym(var_min)), or a string, UQ(var_min).
